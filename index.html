Corridor API Documentation

Overview

 

Corridor API is a programming interface to the Corridor business logic and to the Corridor back-end Oracle database. The API is distributed as a Win32 lib file named CorridorAPI.lib which contains dynamically-linked libraries for the APIs and the Corridor business logic. Supporting header files for C++ classes and other define files are included in the distribution as well. Interfacing with the APIs is through linking the library with an application. Applications interfacing with the APIs do not require MFC. The APIs do not have any UI aspects of their own.

 

The Corridor API can be used with Java applications as well. A Java wrapper through the Java Native Interface is also provided for the APIs. The process of using the APIs with the Java applications is well described through the example code and in further paragraphs.

 

Conventions

 

Unless otherwise stated, all String type data is 255 characters in length. When a data object is described as "Must be valid" this means that the value you are entering must exist in the database in the proper tables that hold objects of the type you are attempting to pass.

 

High-level Design

 

The APIs are designed to be generic and process oriented. They work with request and response objects. A request object contains a request type and a request data payload. The request type is string data and the request payload is a group of primitive data types like string, int, long, boolean, etc. The request object is processed and response object is returned in a synchronous manner. A response object contains a success or failure status flag, response data and/or error data. The response data is similar to the request payload. It is a collection of primitive data. The error data reported consists of one or more errors. Each error has an error code and error text. Session Oriented

 

Interacting with APIs is session oriented. A session is initiated via a login process. The database schema name, user name and user password is required for the login, which authenticates the user and establishes the user authorizations based on the permissions setup for the user. Permissions are established at the initiation of the session. Any changes made to a user's permissions while the session is still active will not be reflected until the current session has ended, and a new session has been initiated. The Corridor API user passwords have distinct encryption. A utility method is provided to generate the passwords. Passwords created in the APIs will not be compatible with passwords created in the Corridor UI unless they are passed through the encryption utility.

 

Transaction Support

 

Within a session one or more requests can be processed. Each request processing verifies the user privileges established during the login. Each request process corresponds to a transaction. On the successful processing of an add/update request, the data is committed. If the request throws an error the data is rolled back. This makes the sessions stateless.

 

Using APIs Through Java

 

The process for using the APIs through a Java Application is via Java wrapper classes that invoke the C and C++ methods of the APIs. The request and response data from the Java objects are translated to the C++ objects  to service the API requests. This is accomplished via a Java native interface DLL file, RitaJNI.dll, created to support such access. You will need to update the PATH environment variable on your system to include the path to the DLL file.

 

The Java wrapper constitutes of a primary class, com.continuum.api.RitaSession, which encapsulates the StartNewSession (), EndSession () and ProcessRequest () methods of the C++ APIs described earlier. Also the Java java.util.Map and java.util.List are used in place of C++ data objects namely, CRitaMap and CRitaArray. API Organization By Module

 

APIs are organized by Corridor module. Each API is used to manipulate, add, and get data that is pertinent to each module inside of Corridor. Use the Contents tab on the left to view all API requests for each listed module.
